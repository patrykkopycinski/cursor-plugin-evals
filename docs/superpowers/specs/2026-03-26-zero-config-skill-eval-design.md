# Zero-Config Skill Eval

> **Goal:** Let a skill author run `cursor-plugin-evals skill-eval init --skill-dir ./my-skill` and get a complete, runnable `eval.yaml` auto-generated from their `SKILL.md` — no manual config needed. After running, the terminal report surfaces 2-3 actionable recommendations. An `--optimize` flag applies those recommendations automatically.

## Problem

Today, writing an `eval.yaml` for a skill requires understanding evaluators, thresholds, adapters, and test design. Skillgrade solves this with a simple flat format, but still requires manual test authoring. We go further: auto-generate the entire eval config from the skill itself.

## User Journey

```
1. Author writes SKILL.md (they already have this)
2. Run: cursor-plugin-evals skill-eval init --skill-dir ./my-skill
   → Reads SKILL.md, generates eval.yaml with 5-8 tests
   → Prints: "Generated eval.yaml with 6 tests. Run: cursor-plugin-evals skill-eval --skill-dir ./my-skill"
3. Run: cursor-plugin-evals skill-eval --skill-dir ./my-skill
   → Executes tests, prints results + recommendations section
4. Optionally: cursor-plugin-evals skill-eval --skill-dir ./my-skill --optimize
   → Runs tests, analyzes results, rewrites eval.yaml with improvements
5. Author tweaks eval.yaml manually, commits it
```

## Components

### 1. Skill Analyzer (`src/skill-init/analyzer.ts`)

Reads `SKILL.md` and produces a structured `SkillProfile`:

```typescript
interface SkillProfile {
  name: string;
  purpose: string;            // one-line description
  capabilities: string[];     // what it can do (e.g., "generate ES|QL queries")
  expectedTools: string[];    // tools it's likely to invoke (empty if pure-text skill)
  keyDomainTerms: string[];   // domain vocabulary for keyword evaluators
  complexity: 'simple' | 'moderate' | 'complex';
  hasCodeOutput: boolean;     // does it produce code/queries/structured output?
  hasFileOutput: boolean;     // does it write files?
}
```

Implementation: single LLM call with structured JSON output. Uses the configured judge model (default `gpt-5.4`). The prompt instructs the LLM to extract the profile from the skill markdown — no tools needed, just text analysis.

If no `SKILL.md` is found in the directory, the command errors with: "No SKILL.md found in <path>. Create a SKILL.md first, then run init again."

### 2. Test Generator (`src/skill-init/generator.ts`)

Takes a `SkillProfile` and generates `GeneratedEval`:

```typescript
interface GeneratedTest {
  name: string;
  prompt: string;
  expected: {
    responseContains?: string[];
    responseNotContains?: string[];
    tools?: string[];
  };
  difficulty: 'simple' | 'moderate' | 'complex';
  category: 'happy-path' | 'edge-case' | 'boundary' | 'negative';
}

interface GeneratedEval {
  name: string;
  description: string;
  evaluators: string[];
  tests: GeneratedTest[];
  defaults: {
    timeout: number;
    repetitions: number;
    thresholds: Record<string, number>;
  };
}
```

**Test distribution:** 3 happy-path, 2 edge-case, 1 boundary, 1 negative (minimum 5, max 8).

**Evaluator auto-selection rules:**
- Always: `correctness`
- If `keyDomainTerms.length > 0`: add `keywords`
- If `hasCodeOutput`: add `script` (with a generated validation script)
- If `expectedTools.length > 0`: add `tool-selection`
- If complexity is `complex`: add `plan-quality`

**Threshold defaults:**
- `correctness`: 0.7
- `keywords`: 0.6
- `script`: 0.5
- `tool-selection`: 0.8

Implementation: single LLM call with structured JSON output. The prompt includes the skill profile and instructions to generate diverse, realistic test cases. The LLM produces the full `GeneratedEval` structure.

### 3. YAML Writer (`src/skill-init/writer.ts`)

Serializes `GeneratedEval` to `eval.yaml` with inline comments:

```yaml
# Auto-generated by cursor-plugin-evals from SKILL.md
# Edit freely — this file is yours to customize.
name: elasticsearch-esql
description: Evaluates ES|QL query generation from natural language

evaluators:
  - correctness    # semantic correctness via LLM judge
  - keywords       # domain term coverage
  - script         # structural validation of generated queries

defaults:
  timeout: 120000
  repetitions: 1
  thresholds:
    correctness: 0.7
    keywords: 0.6
    script: 0.5

tests:
  # Happy path: basic query generation
  - name: basic-from-query
    prompt: "Write an ES|QL query to retrieve all logs from the logs-* index"
    expected:
      response_contains:
        - "FROM"
        - "logs-*"

  # Edge case: aggregation with grouping
  - name: grouped-aggregation
    prompt: "Count logs by severity level"
    expected:
      response_contains:
        - "STATS"
        - "BY"
```

If `eval.yaml` already exists, the command prints an error and exits (no silent overwrite). Use `--force` to overwrite.

### 4. Recommendation Engine (`src/skill-init/recommendations.ts`)

Analyzes `RunResult` and produces `Recommendation[]`:

```typescript
interface Recommendation {
  type: 'evaluator' | 'threshold' | 'test' | 'config';
  priority: 'high' | 'medium' | 'low';
  message: string;           // human-readable explanation
  action?: EvalYamlPatch;    // machine-applicable change (for --optimize)
}

interface EvalYamlPatch {
  op: 'add_evaluator' | 'remove_evaluator' | 'set_threshold' | 'add_test' | 'set_repetitions';
  path: string;              // dot-path in eval.yaml
  value: unknown;
}
```

**Two-phase recommendation pipeline:**

**Phase 1 — Deterministic rules (fast, free):**

| Condition | Recommendation |
|-----------|---------------|
| All tests pass at 1.0 | "Tests may be too easy — consider adding harder edge cases" |
| Any evaluator avg < 0.3 | "Evaluator X scores very low — consider lowering threshold or improving skill" |
| Pass rate 100% with repetitions=1 | "Add `repetitions: 5` to measure consistency (enables pass@k metrics)" |
| `keywords` evaluator not present but tests have `response_contains` | "Consider adding the `keywords` evaluator for automated keyword checking" |
| `script` evaluator not present but skill produces code | "Consider adding a `script` evaluator with structural validation" |
| High variance across tests (stddev > 0.3) | "Skill output is inconsistent — consider adding constraints to your SKILL.md" |
| < 5 tests | "Add more tests for better coverage" |

**Phase 2 — LLM-powered analysis (richer, context-aware):**

After deterministic rules, an LLM call receives the `SKILL.md` content (read from `--skill-dir`), the full `RunResult` (test names, scores, evaluator results, failures), and the current `eval.yaml` content. It produces domain-specific recommendations:

- Coverage gaps: "Your ES|QL skill tests don't cover DISSECT or GROK patterns"
- Test quality: "Test 'basic-query' is too vague — specify the expected index pattern"
- Evaluator suggestions: "Add a script grader that validates SQL syntax for code-producing tests"
- Threshold tuning: "Correctness threshold 0.7 is too strict for this skill's complexity — try 0.6"
- New test ideas: concrete test prompts with expected outputs

The LLM returns structured JSON (`Recommendation[]`), same interface as Phase 1. Both phases merge into a single list, deduplicated by message similarity. Phase 2 is skipped if `--no-llm-recommendations` is passed (for cost-sensitive CI environments).

**Terminal output:** Top 3-5 recommendations (mixed from both phases) printed in a "Recommendations" section after the test results. Always shown.

**--optimize behavior:** Reads `eval.yaml`, applies all `EvalYamlPatch` actions from both phases, rewrites the file, prints what changed.

### 5. CLI Integration

Extends the existing `skill-eval` command:

```
cursor-plugin-evals skill-eval init --skill-dir <path> [--force] [--model <model>]
cursor-plugin-evals skill-eval --skill-dir <path> [--optimize] [...existing opts]
```

- `init`: new subcommand, generates eval.yaml
- `--optimize`: new flag on existing command, applies recommendations after run
- `--force`: overwrite existing eval.yaml during init
- `--model`: override judge model for generation (default: configured judge model)

### Integration with Existing Code

- **Skill loader** (`src/layers/skill/loader.ts`): unchanged — it already loads eval.yaml
- **Skill runner** (`src/layers/skill/index.ts`): unchanged — it already runs skill suites
- **Terminal reporter** (`src/reporting/terminal.ts`): add `printRecommendations()` call after existing report sections
- **RunResult** (`src/core/types.ts`): add optional `recommendations?: Recommendation[]`

### Testing Strategy

- **analyzer.test.ts**: mock LLM response, verify SkillProfile extraction from sample SKILL.md
- **generator.test.ts**: mock LLM response, verify test distribution (categories, evaluator selection rules)
- **writer.test.ts**: unit test YAML serialization, verify comments present, verify round-trip (write → load → matches)
- **recommendations.test.ts**: deterministic — no mocks needed. Feed various RunResult shapes, verify correct recommendations
- **CLI integration**: verify `init` creates file, `--optimize` modifies file, `--force` overwrites

### Cost

- `init` command: 2 LLM calls (analyzer + generator) ≈ $0.01-0.03 depending on model
- `--optimize`: 1 LLM call (recommendation analysis) ≈ $0.01-0.02
- Regular `skill-eval` run: same as today + 1 LLM call for recommendations ≈ $0.01
- Total zero-to-first-optimize: ~$0.05 + eval run cost

### Out of Scope

- Docker-per-trial isolation (separate feature)
- Auto-generating tests without any SKILL.md (requires skill to exist)
- Re-generating tests on every run (tests are materialized, edited manually)
- Fully autonomous optimization loops (single pass per --optimize invocation)
